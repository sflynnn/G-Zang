## 1. 核心运维流程

### 1.1 环境体系

| 环境 | 用途 | 部署策略 | 触发方式 |
|------|------|---------|---------|
| `dev` | 开发自测 | 手动触发 | `git push` |
| `test` | 集成测试 | CI 自动触发 | PR 合并 |
| `staging` | 预发布验证 | CI 自动触发 | 主干合并 |
| `production` | 正式生产 | 需审批 | 主干合并 + 审批 |

### 1.2 部署流程

```
代码提交 → 安全扫描 → 单元测试 → 构建镜像 → 集成测试
    → 部署预发布 → E2E 测试 → 审批（生产）→ 蓝绿部署 → 健康检查
```

### 1.3 回滚流程

```bash
# 查看最近部署版本
kubectl rollout history deployment/gzang-server -n production

# 回滚到上一版本
kubectl rollout undo deployment/gzang-server -n production

# 回滚到指定版本
kubectl rollout undo deployment/gzang-server -n production --to-revision=3
```

## 2. 关键监控指标

| 指标 | 阈值 | 告警级别 |
|------|------|---------|
| HTTP 错误率（5xx） | > 5% | 严重 |
| P95 响应时间 | > 2 秒 | 警告 |
| JVM 堆内存使用率 | > 85% | 警告 |
| 数据库连接池使用率 | > 90% | 严重 |
| Pod 重启次数 | > 3 次/小时 | 警告 |

---

## 3. 紧急响应

| 事件 | 响应时间 | 处理流程 |
|------|---------|---------|
| 服务不可用（P0） | 15 分钟内响应 | 1. 查看 Pod 状态 → 2. 检查日志 → 3. 回滚或扩容 |
| 数据库连接失败 | 15 分钟内响应 | 1. 检查连接池 → 2. 检查网络 → 3. 联系 DBA |
| 大规模报错 | 30 分钟内响应 | 1. 紧急开关降级 → 2. 定位问题 → 3. 发布修复 |
