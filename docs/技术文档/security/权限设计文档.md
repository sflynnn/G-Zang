## 一、设计背景与目标

### 1.1 业务场景

G-Zang（归藏）是一个同时服务于**个人用户**和**企业用户**的财务管理系统：

- **个人用户**：自主记账，数据仅自己可见
- **企业用户**：多人协作记账，包含管理员、财务、普通员工等多种角色

### 1.2 设计目标

1. 支持 **RBAC 角色权限控制**，角色可自定义
2. 支持 **数据范围隔离**，实现多租户数据安全
3. 预留 **字段权限** 和 **行级权限** 扩展点
4. 所有操作均需 **审计日志** 记录
5. 个人用户与企业用户在权限层面统一建模

### 1.3 核心设计原则

- **最小权限原则**：默认无权限，按需授予
- **纵深防御**：前端防君子，后端防小人，双重校验
- **可审计可追溯**：所有操作留痕
- **可扩展**：预留字段权限、行级权限等扩展点

## 二、权限模型

### 2.1 统一建模：个人用户 = 无公司的"企业成员"

将**个人用户**也视为**一个虚拟公司的成员**进行统一建模：

```
┌──────────────────────────────────────────────────────┐
│                                                       │
│   个人用户                                             │
│   ├── company_id = NULL                              │
│   ├── 数据范围 = OWN（仅本人数据）                     │
│   └── 角色 = USER                                     │
│                                                       │
│   企业用户                                             │
│   ├── company_id = 某公司ID                           │
│   ├── 数据范围 = COMPANY（本公司数据）                  │
│   └── 角色 = ADMIN / FINANCE / USER / 自定义角色       │
│                                                       │
└──────────────────────────────────────────────────────┘
```

> **好处**：代码层面无需区分个人/企业，统一用 `company_id` + `data_scope` 逻辑处理。个人用户的"公司"即为其自己。

## 三、权限维度设计

本系统实现以下权限维度：

| 维度 | 说明 | 本期实现 | 预留扩展 |
|------|------|---------|---------|
| 功能权限 | 用户能否使用某个功能/菜单/API | ✅ 实现 | — |
| 数据范围权限 | 用户能查看/操作哪些数据范围 | ✅ 实现 | — |
| 接口权限 | 用户能否调用某个具体接口 | ✅ 实现 | — |
| 字段权限 | 用户能看到/编辑哪些字段 | ❌ 不做 | ✅ 预留 |
| 行级权限 | 基于特定条件过滤数据行 | ❌ 不做 | ✅ 预留 |

## 四、角色体系

### 4.1 内置角色（预置，不可删除）

| 角色名 | 角色编码 | 数据范围 | 说明 |
|--------|---------|---------|------|
| 超级管理员 | `SUPER_ADMIN` | ALL | 系统最高权限，跨租户访问 |
| 企业管理员 | `ADMIN` | COMPANY | 企业账号管理员 |
| 财务 | `FINANCE` | COMPANY | 企业财务人员 |
| 普通用户 | `USER` | OWN | 基础记账功能 |

**内置角色权限矩阵**：

| 权限项 | SUPER_ADMIN | ADMIN | FINANCE | USER |
|--------|:-----------:|:-----:|:--------:|:----:|
| 系统配置 | ✅ | ❌ | ❌ | ❌ |
| 用户管理（企业） | ✅ | ✅ | ❌ | ❌ |
| 公司管理 | ✅ | ✅ | ❌ | ❌ |
| 角色管理（企业） | ✅ | ✅ | ❌ | ❌ |
| 交易录入 | ✅ | ✅ | ✅ | ✅ |
| 交易查看 | ✅ | ✅ | ✅ | ✅ |
| 交易编辑 | ✅ | ✅ | ✅ | ❌ |
| 交易删除 | ✅ | ✅ | ✅ | ❌ |
| 账户管理 | ✅ | ✅ | ✅ | ✅ |
| 分类管理 | ✅ | ✅ | ✅ | ✅ |
| 报表查看 | ✅ | ✅ | ✅ | ❌ |
| 成本中心管理 | ✅ | ✅ | ❌ | ❌ |
| 业务订单管理 | ✅ | ✅ | ✅ | ❌ |

### 4.2 自定义角色（企业可创建）

企业管理员可为本公司创建自定义角色，灵活组合功能权限和数据范围。

**约束**：
- 自定义角色的数据范围 **不能超过** `COMPANY`
- 企业管理员不能授予超出自己权限范围的权限
- 自定义角色不能分配系统级权限（如 `SUPER_ADMIN` 专属权限）

**示例自定义角色**：

| 角色名 | 角色编码 | 数据范围 | 权限 |
|--------|---------|---------|------|
| 成本中心主管 | `COST_CENTER_MANAGER` | DEPARTMENT | 交易查看/编辑 + 账户查看 |
| 只读审计员 | `AUDITOR_READONLY` | COMPANY | 交易查看 + 报表查看 |

## 五、数据范围体系

### 5.1 数据范围枚举（DataScope）

```
┌─────────────────────────────────────────────────────┐
│  DataScope                                          │
│                                                     │
│  ├── OWN        — 仅本人数据（user_id = current）  │
│  ├── DEPARTMENT  — 本部门/成本中心数据（预留）       │
│  ├── COMPANY     — 本公司全部数据                   │
│  └── ALL         — 全部数据（超级管理员）            │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 5.2 数据范围执行规则

```
查询请求进入
    │
    ▼
获取用户的数据范围码（DataScope）
    │
    ├── OWN        → WHERE user_id = #{currentUserId}
    ├── DEPARTMENT → WHERE department_id = #{currentUser.departmentId}
    ├── COMPANY    → WHERE company_id = #{currentCompanyId}
    └── ALL        → 无额外条件（查全部）
```

### 5.3 数据范围与角色的默认映射

| 角色 | 默认数据范围 | 说明 |
|------|:----------:|------|
| SUPER_ADMIN | ALL | 跨租户，不受限制 |
| ADMIN | COMPANY | 管理员看本公司全部 |
| FINANCE | COMPANY | 财务看本公司全部 |
| USER | OWN | 普通用户只看自己 |
| 自定义角色 | COMPANY 或更窄 | 由创建者指定 |

## 七、数据模型设计

### 7.1 现有表结构

```
t_user          — 用户表（已有 role_id, company_id 字段）
t_role          — 角色表（已有 role_name, role_code, description）
t_permission    — 权限表（已有 permission_name, permission_code）
t_role_permission — 角色-权限关联表（已有）
```

### 7.2 扩展需求

#### 7.2.1 角色表扩展（t_role）

| 字段 | 类型 | 说明 |
|------|------|------|
| `role_type` | TINYINT | 角色类型：1=系统内置, 2=企业自定义 |
| `data_scope` | VARCHAR(32) | 数据范围：OWN / DEPARTMENT / COMPANY / ALL |
| `company_id` | BIGINT | 自定义角色所属公司（内置角色为 NULL） |
| `is_default` | TINYINT | 是否为某角色的默认角色（预留） |

#### 7.2.2 用户-角色关联表（t_user_role）

```sql
CREATE TABLE t_user_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    role_id BIGINT NOT NULL COMMENT '角色ID',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_role (user_id, role_id),
    INDEX idx_user_id (user_id),
    INDEX idx_role_id (role_id)
) COMMENT='用户角色关联表';
```

#### 7.2.3 角色数据范围关联表（t_role_data_scope）

```sql
CREATE TABLE t_role_data_scope (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_id BIGINT NOT NULL COMMENT '角色ID',
    scope_type VARCHAR(32) NOT NULL COMMENT '范围类型：DEPARTMENT/COST_CENTER/AREA',
    scope_value VARCHAR(128) NOT NULL COMMENT '范围值：如部门ID、成本中心ID',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_role_id (role_id)
) COMMENT='角色数据范围关联表（行级权限预留）';
```

#### 7.2.4 字段权限表（t_field_permission）— 预留

```sql
CREATE TABLE t_field_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_id BIGINT NOT NULL COMMENT '角色ID',
    entity_type VARCHAR(64) NOT NULL COMMENT '实体类型：如Transaction, Account',
    field_name VARCHAR(64) NOT NULL COMMENT '字段名',
    field_visible TINYINT NOT NULL DEFAULT 1 COMMENT '是否可见：0=不可见, 1=可见',
    field_editable TINYINT NOT NULL DEFAULT 1 COMMENT '是否可编辑：0=只读, 1=可编辑',

    INDEX idx_entity_type (entity_type)

```

## 九、鉴权组件设计

### 9.1 组件架构

```
┌─────────────────────────────────────────────────────────────┐
│                     请求进入                                 │
│                      ↓                                      │
│           JwtAuthenticationFilter                          │
│           ├── 解析 JWT Token                                │
│           ├── 加载用户信息 + 角色 + 权限码                    │
│           ├── 设置 SecurityContext                          │
│           └── 设置 TenantContextHolder                      │
│                      ↓                                      │
│         SecurityFilterChain（URL 白名单校验）                 │
│                      ↓                                      │
│         MethodSecurityInterceptor（方法级权限注解拦截）        │
│         ├── @RequirePermission("xxx") — 功能权限            │
│         ├── @RequireDataScope — 数据范围                    │
│         └── @PreAuthorize — Spring Security 原生           │
│                      ↓                                      │
│            TenantInterceptor（MyBatis 层）                  │
│            └── 自动注入 company_id 过滤条件                  │
│                      ↓                                      │
│           OperationLogAspect（AOP 切面）                    │
│           └── 记录操作日志                                  │
└─────────────────────────────────────────────────────────────┘
```

### 9.2 核心注解设计

#### 功能权限注解

```java
@RequirePermission("TRANSACTION_ADD")
// 支持多权限（AND 关系）
@RequirePermission({"TRANSACTION_VIEW", "TRANSACTION_EDIT"})
// 支持 OR 关系
@RequirePermission(value = {"TRANSACTION_DELETE", "TRANSACTION_EDIT"}, logical = Logical.OR)
```

#### 数据范围注解

```java
// 数据范围由用户角色决定，自动注入，无需手动标注
@RequireDataScope(DataScope.COMPANY)
```

#### 操作日志注解

```java
@OperationLog(module = "TRANSACTION", action = "CREATE",
             describe = "录入交易记录：#{#dto.amount}元")
public Result<TransactionVO> create(@RequestBody CreateTransactionDTO dto) { ... }
```

### 9.3 权限加载与缓存

**流程**：

```
用户登录 → 查询用户角色 → 查询角色权限码 → 查询角色数据范围
       → 组装 UserDetails（含权限列表） → 生成 JWT（含 userId/roleId/companyId）
       → 存入 Redis（key=permission:user:{userId}，TTL=token有效期）
```

**缓存策略**：
- Redis Key：`g:permission:user:{userId}`
- Value：`{permissions: [...], dataScope: "COMPANY", roleId: 2}`
- 过期时间：与 JWT 过期时间一致
- 更新策略：角色/权限变更时删除缓存，强制重新加载

## 十、数据范围执行机制

### 10.1 多层防护

```
┌──────────────────────────────────────────────────────────┐
│  第一层：MyBatis TenantInterceptor                        │
│  └── 自动注入 company_id 条件（已有）                     │
│                                                           │
│  第二层：Service 层数据范围过滤（兜底）                    │
│  └── 根据 DataScope 动态拼接 WHERE 条件                    │
│                                                           │
│  第三层：MyBatis 拦截器数据范围注入                        │
│  └── 根据注解动态注入数据范围 SQL 片段                     │
└──────────────────────────────────────────────────────────┘
```

### 10.2 数据范围与操作对照

| 操作 | OWN | DEPARTMENT | COMPANY | ALL |
|------|:---:|:----------:|:-------:|:---:|
| SELECT（查列表） | ✅ 仅本人 | ✅ 本部门 | ✅ 本公司 | ✅ 全部 |
| SELECT（查详情） | ✅ 仅本人 | ✅ 本部门 | ✅ 本公司 | ✅ 全部 |
| INSERT（新增） | ✅ 创建者=本人 | ✅ 创建者=本人 | ✅ 本公司 | ✅ 全部 |
| UPDATE（编辑） | ✅ 仅本人创建 | ✅ 仅本部门 | ✅ 本公司 | ✅ 全部 |
| DELETE（删除） | ✅ 仅本人创建 | ✅ 仅本部门 | ✅ 本公司 | ✅ 全部 |

## 十一、审计日志设计

### 11.1 记录时机

| 操作类型 | 触发条件 |
|---------|---------|
| `CREATE` | 新增数据成功时 |
| `UPDATE` | 更新数据成功时 |
| `DELETE` | 删除数据成功时 |
| `VIEW` | 查看敏感数据时（如导出、查看他人敏感信息） |

## 十二、扩展点预留

### 12.1 字段权限扩展

**启用条件**：当 `t_field_permission` 表有对应记录时，Controller 层根据当前用户角色，过滤掉不可见/不可编辑的字段。

**前端配合**：API 返回字段时附带 `editable` / `visible` 标记，前端据此动态渲染表单。

### 12.2 行级权限扩展

**启用条件**：当 `t_role_data_scope` 表有对应记录时，在 SQL 查询时动态注入额外条件（如 `AND cost_center_id IN (...)`）。

**实现方式**：MyBatis 拦截器根据用户角色，动态追加 WHERE 条件片段。
